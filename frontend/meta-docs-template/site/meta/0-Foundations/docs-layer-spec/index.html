    Documentation Layer Specification¶

    Defines the canonical structure, requirements, and compliance criteria for the /docs layer in AI-native, vibe-coding projects.

    Purpose¶

    Establish a standardized, AI-readable documentation layer that supports the complete development lifecycle from product planning through debugging and maintenance.

    Canonical Folder Tree¶

    /docs/

 0-Overview/                     # Project overview and standards
    README.md                   # Project introduction and navigation
    glossary.md                 # Project-specific terminology
    architecture-overview.md    # High-level system architecture
    coding-standards.md         # Development standards and conventions

 1-Product/                      # Product planning and requirements
    roadmap.md                  # Product roadmap and milestones
    prds/                       # Product Requirements Documents
        [feature-name]-prd.md   # Individual feature PRDs

 2-Design/                       # Technical design and architecture
    system-architecture.md      # Detailed system design
    data-models.md              # Data structures and schemas
    api-spec.md                 # API specifications and contracts
    frontend.md                 # Frontend architecture and components
    backend.md                  # Backend services and infrastructure
    integrations.md             # External integrations and APIs

 3-Development/                  # Development process and tracking
    changelog.md                # Version history and changes
    sprint-log.md               # Sprint planning and retrospectives
    architecture-decisions.md   # ADRs and technical decisions
    backlog.md                  # Feature backlog and prioritization
    milestones.md               # Project milestones and deadlines
    release-checklist.md        # Release process and validation
    release-notes/              # Release documentation
        v[version]-notes.md     # Individual release notes

 4-Debugging/                    # Debugging and troubleshooting
    error-log.md                # Error tracking and resolution
    known-issues.md             # Known issues and workarounds
    debug-playbook.md           # Debugging procedures and tools

 5-Prompts/                      # AI prompts and session notes
     frontend-prompts.md         # Frontend development prompts
     backend-prompts.md          # Backend development prompts
     debugging-prompts.md        # Debugging and troubleshooting prompts
     refactor-prompts.md         # Code refactoring prompts
     session-notes/              # AI collaboration session notes
         [date]-session.md       # Individual session documentation

    File-Level Requirements (per folder)¶

    0-Overview/¶

    • README.md - Project introduction, quick start, and navigation guide
    • glossary.md - Project-specific terms, acronyms, and definitions
    • architecture-overview.md - High-level system architecture and component relationships
    • coding-standards.md - Development standards, conventions, and best practices

    1-Product/¶

    • roadmap.md - Product vision, milestones, and timeline
    • prds/ - Individual Product Requirements Documents following PRD template

    2-Design/¶

    • system-architecture.md - Detailed technical architecture and design decisions
    • data-models.md - Database schemas, data structures, and models
    • api-spec.md - API endpoints, contracts, and integration specifications
    • frontend.md - Frontend architecture, components, and user interface design
    • backend.md - Backend services, infrastructure, and business logic
    • integrations.md - External service integrations and third-party APIs

    3-Development/¶

    • changelog.md - Version history, changes, and release tracking
    • sprint-log.md - Sprint planning, retrospectives, and team coordination
    • architecture-decisions.md - Architecture Decision Records (ADRs) and technical choices
    • backlog.md - Feature backlog, prioritization, and task management
    • milestones.md - Project milestones, deadlines, and progress tracking
    • release-checklist.md - Release process, validation steps, and deployment procedures
    • release-notes/ - Individual release documentation and announcements

    4-Debugging/¶

    • error-log.md - Error tracking, resolution history, and patterns
    • known-issues.md - Known issues, workarounds, and resolution status
    • debug-playbook.md - Debugging procedures, tools, and troubleshooting guides

    5-Prompts/¶

    • frontend-prompts.md - AI prompts for frontend development tasks
    • backend-prompts.md - AI prompts for backend development tasks
    • debugging-prompts.md - AI prompts for debugging and troubleshooting
    • refactor-prompts.md - AI prompts for code refactoring and improvement
    • session-notes/ - AI collaboration session documentation and outcomes

    Naming Conventions¶

    • Files: kebab-case with descriptive names (e.g., system-architecture.md)
    • Folders: numbered prefixes with descriptive names (e.g., 0-Overview/)
    • PRDs: [feature-name]-prd.md (e.g., user-authentication-prd.md)
    • Release notes: v[version]-notes.md (e.g., v1.2.0-notes.md)
    • Session notes: [date]-session.md (e.g., 2024-12-19-session.md)

    Update Rules (reference vs working docs)¶

    Reference Documents (Infrequent Updates)¶

    • 0-Overview/ - Update when project scope or standards change
    • 2-Design/ - Update when architecture or technical approach changes
    • 4-Debugging/ - Update when new debugging procedures or issues are identified

    Working Documents (Frequent Updates)¶

    • 1-Product/ - Update with each feature or requirement change
    • 3-Development/ - Update continuously during development cycles
    • 5-Prompts/ - Update with each AI collaboration session

    Cross-Links (what must link to what)¶

    Required Cross-References¶

    • PRDs Design docs - Requirements must link to technical design
    • Design docs API spec - Architecture must link to API contracts
    • Changelog Release notes - Version history must link to release documentation
    • Architecture decisions System architecture - ADRs must link to design documents
    • Debug playbook Error log - Debugging procedures must link to error history

    Optional Cross-References¶

    • Roadmap Milestones - Product planning links to development tracking
    • Backlog Sprint log - Feature planning links to sprint execution
    • Session notes Prompts - AI collaboration links to prompt templates

    Acceptance Criteria (what "compliant docs layer" means)¶

    A compliant /docs layer must have:

    1. Complete Structure - All required folders and files present
    2. Proper Naming - Files follow established naming conventions
    3. Required Cross-Links - All mandatory cross-references implemented
    4. Front-Matter - All files include proper YAML front-matter
    5. Purpose Statements - All files have clear purpose and audience definitions
    6. AI Readability - All content structured for AI parsing and generation
    7. Version Control - All files tracked in version control
    8. Update Tracking - All files include update and review information
